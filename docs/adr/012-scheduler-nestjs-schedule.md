# ADR-012: Планировщик синхронизации источников: @nestjs/schedule + @Cron

**Статус:** Принято
**Дата:** 2026-03-24

---

## Контекст

Для периодического обхода RSS-лент и Telegram-каналов нужен планировщик. Задача: каждые N минут обойти все активные источники, скачать новые статьи и поставить их в очередь на AI-оценку. Требования: не запускать следующий цикл пока не завершился предыдущий (чтобы не накапливать параллельные запуски при медленной синхронизации), контролировать параллелизм запросов к внешним API.

---

## Решение

`@nestjs/schedule` с декоратором `@Cron` + `pLimit` для ограничения параллелизма:

```typescript
import { Cron, CronExpression } from '@nestjs/schedule';
import pLimit from 'p-limit';

@Cron(CronExpression.EVERY_30_MINUTES)
async syncAllSources(): Promise<void> {
  const limit = pLimit(5); // не более 5 источников параллельно
  await Promise.all(sources.map(s => limit(() => this.syncSource(s))));
}
```

- `@Cron` не запускает следующую итерацию пока не завершилась предыдущая — в отличие от `setInterval`, который накапливает вызовы
- `pLimit(5)` ограничивает одновременные запросы к внешним API, снижая риск rate-limit
- Интервал вынесен в env-переменную `FEED_SYNC_INTERVAL_MIN` (default: 30)

---

## Альтернативы

| Вариант                         | Почему отклонён                                                                                                                    |
| ------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| `setInterval` в `onModuleInit`  | Накапливает параллельные вызовы если предыдущий цикл не успел завершиться; нет механизма предотвращения overlap                    |
| BullMQ / очереди задач          | Избыточно: требует Redis, отдельного воркера, управления очередью — при том что задача не требует персистентности между рестартами |
| Отдельный cron-процесс / сервис | Усложняет деплой (ещё один контейнер), нет смысла при монолитной архитектуре                                                       |
| `@nestjs/bull`                  | Те же минусы что у BullMQ — Redis как новая зависимость ради одной задачи                                                          |

---

## Последствия

**Плюсы:**

- Встроен в NestJS-экосистему, нет внешних зависимостей кроме самого пакета
- `@Cron` гарантирует отсутствие параллельных запусков одного и того же задания
- `pLimit` даёт точный контроль над concurrency без сложности очередей

**Минусы:**

- Состояние планировщика не персистентно: после рестарта контейнера следующий запуск — через полный интервал (не «через сколько осталось»)
- При горизонтальном масштабировании (несколько инстансов API) каждый инстанс запускает свой планировщик — нет distributed lock. Сейчас не актуально, но при масштабировании потребует координации (например, через Redis lock или вынос в отдельный воркер)
