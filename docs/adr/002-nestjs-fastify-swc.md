# 002. NestJS: Fastify-адаптер + SWC-компилятор

**Статус:** Принято

**Дата:** 2026-02-28

## Контекст

NestJS поддерживает два HTTP-адаптера (Express и Fastify) и два компилятора (tsc и SWC). Выбор влияет на производительность в рантайме и скорость сборки/перезапуска в dev-режиме.

## Рассмотренные варианты

### HTTP-адаптер

**Express (по умолчанию)**
- ✅ Самый популярный, огромная экосистема middleware
- ✅ Большинство туториалов и примеров написаны под Express
- ❌ Медленнее Fastify на ~20-30% по throughput в бенчмарках
- ❌ Менее строгая типизация request/response

**Fastify** ✓
- ✅ Быстрее Express, особенно на большом количестве небольших запросов
- ✅ Встроенная JSON-схема-валидация, лучшая производительность JSON-сериализации
- ✅ Строгая типизация через TypeScript плагины
- ❌ Не все Express-middleware совместимы (нужны Fastify-обёртки)
- ❌ Меньше примеров в документации NestJS

### Компилятор

**tsc (TypeScript compiler)**
- ✅ Стандарт, полная совместимость
- ❌ Медленный в watch-режиме (полная перекомпиляция)
- ❌ Не поддерживает decorator metadata нативно без дополнительных плагинов

**SWC** ✓
- ✅ В 10-20x быстрее tsc при инкрементальной сборке
- ✅ Официально поддерживается в `@nestjs/cli` через `nest-cli.json`
- ✅ Поддерживает `emitDecoratorMetadata` через `.swcrc`
- ❌ Не выполняет type-checking (только транспиляция) — typecheck остаётся за `tsc --noEmit`

## Решение

Выбраны **Fastify + SWC**.

Fastify даёт выигрыш в производительности без существенных ограничений для типичного REST API. SWC критически ускоряет inner dev loop: перезапуск при изменении файла занимает <1 секунды вместо 5-10.

Type-checking при этом не отключён — он вынесен в отдельную задачу (`pnpm typecheck`) и выполняется в CI.

## Последствия

### Положительные
- Быстрый перезапуск в dev-режиме (SWC)
- Более высокая пропускная способность в продакшене (Fastify)
- `nest start --watch` практически мгновенен

### Отрицательные / компромиссы
- Express-middleware нельзя подключить напрямую — нужна Fastify-обёртка или замена
- SWC не проверяет типы: ошибки TypeScript обнаруживаются только через `pnpm typecheck` или IDE
- Fastify требует явного указания типов для `request.body` (нет автоматического inference из `@Body()`)

### Риски
- При использовании сторонних NestJS-плагинов, написанных только для Express, потребуется ручная адаптация
