# ADR-011: JSONB-поле для хранения пользовательских настроек

**Статус:** Принято
**Дата:** 2026-03-24

---

## Контекст

Настройки фильтрации пользователя включают несколько взаимосвязанных параметров: порог релевантности, текст интересов, набор категорий. Кроме того, каждый из этих параметров можно переопределить отдельно для каждого источника (per-source). Изначально настройки хранились в двух местах: `User.relevanceThreshold` (отдельное поле) и таблица `UserPreference` (M2M для категорий). При добавлении per-source переопределений это потребовало бы либо дублирования структуры, либо сложной нормализации.

---

## Решение

Единая таблица `UserPreferences` с полем `settings Json`:

```prisma
model UserPreferences {
  id       String  @id @default(cuid())
  userId   String
  sourceId String? // null = глобальные, значение = per-source

  settings Json @default("{}")
  // Структура settings:
  //   relevanceThreshold?: number      // default 0.6
  //   interestsText?: string | null
  //   selectedCategories?: string[]    // slugs

  @@unique([userId, sourceId])
}
```

- `sourceId = null` — глобальные настройки пользователя
- `sourceId = <id>` — переопределение для конкретного источника
- При оценке статьи: per-source настройки имеют приоритет над глобальными

Типизация `settings` обеспечивается на уровне приложения через Zod-схему в `@repo/shared`.

---

## Альтернативы

| Вариант                                                                                               | Почему отклонён                                                                              |
| ----------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------- |
| Отдельные колонки для каждого параметра (`relevanceThreshold`, `interestsText`, `selectedCategories`) | При добавлении нового параметра требует миграции схемы; per-source дублирование всех колонок |
| Нормализованные таблицы (отдельная строка на каждый параметр)                                         | Избыточная сложность для небольшого фиксированного набора настроек; сложные JOIN при чтении  |
| Две таблицы: глобальные настройки + per-source                                                        | Дублирование структуры, рассинхронизация при изменении набора параметров                     |

---

## Последствия

**Плюсы:**

- Глобальные и per-source настройки хранятся одной строкой с одним `UNIQUE([userId, sourceId])` — нет дублирования структуры
- Добавление нового параметра не требует миграции схемы, только изменения Zod-схемы и бизнес-логики
- Простое чтение: один `findUnique` вместо нескольких JOIN

**Минусы:**

- Нет constraints на уровне БД (нельзя проверить что `relevanceThreshold` в диапазоне 0–1) — валидация только на уровне приложения
- При сложных запросах по значениям внутри JSON (`settings->>'relevanceThreshold' > 0.8`) потребуется составной индекс
- Рефакторинг поля требует data migration (JSON не версионируется автоматически)
