# ADR-010: Gate — слой интеграции с внешними API

**Статус:** Принято
**Дата:** 2026-03-24

---

## Контекст

При интеграции с внешними HTTP-сервисами (Groq, в будущем — Telegram, push-уведомления и т.д.) логика транспорта смешивается с бизнес-логикой: retry, rate-limit, парсинг HTTP-ошибок оказываются в тех же классах, что строят промпты или обрабатывают данные. Это затрудняет тестирование и повторное использование.

---

## Решение

Ввести тип классов **Gate** — тонкая обёртка над одним внешним HTTP-API.

### Ответственность Gate

| Входит в Gate                                    | Не входит в Gate                          |
| ------------------------------------------------ | ----------------------------------------- |
| Инициализация SDK / HTTP-клиента                 | Бизнес-логика (построение промптов)       |
| Отправка запроса и получение сырого ответа       | Парсинг доменных объектов из ответа       |
| Retry при 429 / 503 с правильной задержкой       | Исключения NestJS (`BadRequestException`) |
| Логирование транспортных ошибок                  | Знание о доменных сущностях               |
| Проверка доступности (ключ API задан / не задан) |                                           |

Gate возвращает **сырой результат или null** — никогда не бросает доменных исключений.

### Соглашения по именованию

```
{Имя сервиса}Gate  →  GroqGate, TelegramGate, SmtpGate
```

Файл: `src/{модуль}/{имя}.gate.ts`

### Пример: GroqGate

```
ScoringService  →  GroqGate  →  Groq SDK  →  Groq API
```

- `GroqGate` — транспорт: отправить сообщения, получить текст, обработать 429
- `ScoringService` — логика: построить промпт, распарсить JSON, вернуть `ScoreResult`

### Граф зависимостей

```
SourcesService  →  ScoringService  →  GroqGate  →  Groq API
```

## Альтернативы

| Вариант                                          | Почему отклонён                                           |
| ------------------------------------------------ | --------------------------------------------------------- |
| Вся интеграция в сервисе                         | Смешивает транспорт с логикой, тяжело тестировать         |
| Отдельный NestJS-модуль на каждый внешний сервис | Избыточно для одного клиента без собственных контроллеров |
| Общий `HttpGateway` для всех внешних вызовов     | Теряет типизацию конкретного API, слишком абстрактно      |

---

## Последствия

**Плюсы:**

- Сервисы тестируются без реального HTTP — достаточно замокать Gate
- Retry / rate-limit логика сосредоточена в одном месте
- Замена SDK (например, переход на другую LLM) затрагивает только Gate

**Минусы:**

- Дополнительный файл и слой косвенности на каждую интеграцию
