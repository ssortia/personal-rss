# 001. Монорепо: pnpm workspaces + Turborepo

**Статус:** Принято

**Дата:** 2026-02-28

## Контекст

Проект состоит из нескольких взаимосвязанных частей: NestJS API, Next.js веб-приложение, общие типы, UI-компоненты, конфигурации. Их нужно разрабатывать, тестировать и деплоить скоординированно.

Основные требования:
- Общие пакеты (`@repo/types`) должны быть доступны без публикации в npm
- Изменение общего типа должно сразу отражаться в API и web без пересборки
- Команды (`lint`, `build`, `test`) должны работать как для всего репо, так и для отдельного пакета
- Кэширование сборок для ускорения CI

## Рассмотренные варианты

**Вариант A: Полирепо (отдельные репозитории)**
- ✅ Полная изоляция, независимые релизы
- ❌ Сложная синхронизация версий общих пакетов
- ❌ Необходима публикация в npm или private registry для шаринга кода
- ❌ Трудно рефакторить across-repo

**Вариант B: Монорепо с npm workspaces + Lerna**
- ✅ Шаринг кода без публикации
- ❌ Lerna устаревает, сообщество перешло на другие инструменты
- ❌ npm workspaces медленнее pnpm по install-времени

**Вариант C: Монорепо с pnpm workspaces + Turborepo** ✓
- ✅ pnpm — самый быстрый пакетный менеджер, строгая изоляция зависимостей
- ✅ Turborepo — умное кэширование, параллельное выполнение задач
- ✅ Активно поддерживается (Vercel)
- ✅ Нативная поддержка workspace-протокола (`workspace:*`)
- ❌ Turborepo требует понимания pipeline-конфигурации

**Вариант D: Монорепо с Nx**
- ✅ Мощные инструменты для code generation
- ❌ Высокий порог входа, избыточная сложность для этого размера проекта
- ❌ Сильная привязка к экосистеме Nx

## Решение

Выбран **Вариант C: pnpm workspaces + Turborepo**.

pnpm выбран за скорость и строгость: он не допускает использование зависимостей, не объявленных в `package.json` пакета, что предотвращает случайные transitive dependency issues.

Turborepo выбран как наименее invasive инструмент оркестрации: конфигурация — один файл `turbo.json`, без генераторов и сложных концепций.

## Последствия

### Положительные
- Единый `pnpm install` для всего репо
- Изменения в `@repo/types` моментально видны в API и web (через `workspace:*` + path aliases)
- CI кэширует результаты задач через `.turbo`
- Команды типа `pnpm --filter @repo/api dev` для точечной работы

### Отрицательные / компромиссы
- Разработчики должны знать pnpm (нельзя использовать npm или yarn)
- Turborepo pipeline нужно держать актуальным при добавлении новых задач
- Глобальные зависимости (devDependencies в корне) могут создавать путаницу

### Риски
- Turborepo активно развивается; breaking changes при мажорном обновлении возможны
