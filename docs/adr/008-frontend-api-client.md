# ADR-008: Архитектура API-клиента на фронтенде

**Статус:** Принято
**Дата:** 2026-03-04

---

## Контекст

В начальной версии фронтенда запросы к бэкенду были организованы стихийно:

- HTTP-пути (`'/users'`, `'/auth/login'`) — строки прямо в компонентах и Server Actions
- Ошибки парсились по строке: `message.includes('409')` вместо проверки статус-кода
- `@tanstack/react-query` установлен, но не подключён — нет `QueryClientProvider`, нет хуков
- `@t3-oss/env-nextjs` установлен, но переменные читались через сырой `process.env[]`
- В `auth.ts` — прямые вызовы `fetch()` с дублированием URL из переменной окружения

С ростом числа эндпоинтов такой подход приводит к дублированию, сложной отладке и невозможности переиспользовать логику между Server и Client Components.

---

## Рассмотренные варианты

- **Оставить как есть** — просто добавить `QueryClientProvider`. Не решает проблему дублирования путей и отсутствия типизированных ошибок.

- **OpenAPI-кодогенерация (orval, hey-api)** — генерирует типизированный клиент из swagger-схемы бэкенда. Мощный инструмент, но избыточен для шаблона: требует CI-шага кодогенерации, усложняет onboarding, связывает фронт с конкретной версией схемы.

- **tRPC** — устраняет границу API полностью. Принципиальное архитектурное изменение; несовместимо с существующим NestJS REST API и next-auth.

- **Трёхуровневая архитектура (выбрано)** — тонкий `fetch`-враппер + доменные функции + React Query хуки. Минимум абстракций, максимум явности.

---

## Решение

Ввести три слоя, разделённые по ответственности:

```
Server Components / Server Actions / auth.ts
            ↓
      src/api/*.api.ts          ← доменные функции (типизированные, без React)
            ↓
         src/lib/api.ts         ← HTTP-клиент (fetch-враппер + ApiError)

Client Components
            ↓
      src/hooks/use-*.ts        ← React Query хуки (оборачивают src/api/*.api.ts)
```

### HTTP-клиент (`src/lib/api.ts`)

Тонкая обёртка над `fetch`. Единственные обязанности:

- Установить заголовки `Content-Type` и `Authorization`
- Бросить `ApiError` (со статус-кодом) при `!res.ok`
- Обработать `204 No Content`

`ApiError` — типизированный класс-ошибка с полем `status: number`. Позволяет в вызывающем коде проверять статус без парсинга строки сообщения.

`API_BASE` выбирается один раз при инициализации модуля: `env.API_URL` на сервере (внутренняя сеть / docker), `env.NEXT_PUBLIC_API_URL` на клиенте.

### Валидация окружения (`src/lib/env.ts`)

`@t3-oss/env-nextjs` c `createEnv`. Переменные валидируются на старте приложения — ошибка конфигурации выявляется сразу, а не в рантайме при первом запросе.

### Доменные функции (`src/api/*.api.ts`)

Чистые `async`-функции без React и без хуков. Знают только URL и тип ответа. Переиспользуются напрямую в Server Components, Server Actions и `auth.ts`.

```ts
// Пример: usersApi.list(accessToken) вместо api.get('/users', { accessToken })
```

Токен передаётся явным аргументом — намеренное решение: доменные функции остаются чистыми и не привязаны ни к сессии, ни к React-контексту.

### React Query хуки (`src/hooks/use-*.ts`)

Обёртки над доменными функциями для Client Components. Сами получают токен из `useSession`. Подписываются на правило: клиентский компонент не вызывает `usersApi.*` напрямую.

```ts
// Клиентский компонент:
const { data: users } = useUsers(); // токен скрыт в хуке

// Server Component:
const users = await usersApi.list(accessToken); // токен явный
```

### Правило использования

| Контекст         | Использует                                              |
| ---------------- | ------------------------------------------------------- |
| Server Component | `*.api.ts` напрямую                                     |
| Server Action    | `*.api.ts` напрямую                                     |
| `auth.ts`        | `*.api.ts` напрямую                                     |
| Client Component | `use-*.ts` хуки                                         |
| Нигде            | `api.get('/users', ...)` — только через доменный модуль |

## Последствия

### Положительные

- Все URL эндпоинтов определены в одном месте — рефакторинг пути затрагивает один файл
- `ApiError.status` даёт надёжную проверку кода ответа вместо парсинга строки
- `@tanstack/react-query` подключён и готов к использованию с правильным паттерном (`useState` + фабрика, не синглтон)
- Env-переменные валидируются при старте, а не в рантайме
- Доменные функции легко тестировать — они не зависят от React

### Отрицательные / компромиссы

- Токен передаётся явным аргументом в каждый метод. При большом числе эндпоинтов это утомляет; альтернатива — фабрика `createApiClient(token)`, захватывающая токен в замыкании
- Два слоя вызовов для клиентских компонентов: хук → доменная функция → HTTP-клиент. Для простых запросов — оверхед косвенности

### Риски

- Если разработчик вызовет `usersApi.*` напрямую в Client Component (вместо хука), токен нужно будет передавать вручную через `useSession` — это не запрещено архитектурой, лишь не рекомендовано
