# ADR-007: Слой репозиториев в API

**Статус:** Принято
**Дата:** 2026-03-04

---

## Контекст

Сервисы (`UsersService`, `AuthService`) напрямую инжектировали `PrismaService` и смешивали SQL-запросы с бизнес-логикой (bcrypt, guards, исключения). Это затрудняет тестирование — чтобы протестировать бизнес-правило, нужно мокать Prisma — и нарушает принцип единственной ответственности.

---

## Решение

Ввести класс `UsersRepository` как единственную точку доступа к БД в домене пользователей.

### Разделение ответственности

| Слой              | Владеет                                            |
| ----------------- | -------------------------------------------------- |
| `UsersRepository` | Prisma-вызовы, SQL-запросы, выбор полей (`select`) |
| `UsersService`    | Бизнес-логика: проверки, исключения                |
| `AuthService`     | Auth-логика: bcrypt, JWT-генерация                 |

### Ключевые решения

**`BaseRepository<TModel, TCreateInput, TUpdateInput>`** — абстрактный базовый класс в `src/common/repository/base.repository.ts`. Содержит пять операций, опирающихся на идентификатор (`findById`, `findAll`, `create`, `update`, `delete`). `findAll` / `create` / `update` / `delete` — `protected`; конкретный репозиторий сам решает, что открыть публично.

**Каст `as unknown as BaseModelDelegate<...>`** — Prisma-делегаты используют сложные условные дженерики (`SelectSubset`, `GetFindResult`, `Prisma__ModelClient`), которые TypeScript не может унифицировать с простым структурным интерфейсом в точке присваивания. Каст изолирован в конструкторе подкласса; все вызовы в базовом классе и доменных методах полностью типобезопасны.

**Два способа доступа к Prisma в подклассе:**

- `this.delegate` (через базовый класс) — для операций, реализованных в базе (`findById` и т.д.)
- `this.prisma.user` (прямой, с полными Prisma-типами) — для доменных запросов с `select`, `orderBy`, `where` по произвольному полю

**`PUBLIC_SELECT` как единственный источник истины** — константа `as const satisfies Prisma.UserSelect` определена один раз в репозитории. `PublicUser` выводится из неё через `Prisma.UserGetPayload`, что исключает расхождение между SQL-запросом и TypeScript-типом.

**Репозиторий не бросает исключения** — только возвращает `null` или выбрасывает ошибки Prisma. Вся доменная логика (`ForbiddenException`, `ConflictException`) остаётся в сервисах.

**`UsersRepository` не экспортируется из `UsersModule`** — это деталь реализации модуля.

**`AuthService` убрал прямую зависимость от `PrismaService`** — делегирует через `usersService.updateRefreshToken(userId, hashedToken | null)`.

### Граф зависимостей

```
AuthService  →  UsersService  →  UsersRepository  →  PrismaService
                                       ↑
                               BaseRepository<T, C, U>
```

## Альтернативы

| Вариант                                            | Почему отклонён                                                                        |
| -------------------------------------------------- | -------------------------------------------------------------------------------------- |
| Оставить Prisma в сервисах                         | Смешивает SQL с логикой; сложнее юнит-тестировать сервисы                              |
| Без базового класса, только конкретные репозитории | Дублирование `findById` / `create` / `delete` в каждом репозитории по мере роста схемы |
| Отдельный Prisma-модуль на каждый домен            | Усложняет структуру без выгоды при текущем масштабе                                    |

---

## Последствия

**Плюсы:**

- Сервисы можно юнит-тестировать, мокая `UsersRepository` вместо Prisma
- Изменение схемы БД затрагивает только репозиторий, не сервис
- `PublicUser` — один тип, один `select`, нет дублирования

**Минусы:**

- Дополнительный файл и слой косвенности для каждого домена
- При сложных запросах (JOIN, агрегации) граница между репозиторием и сервисом может размыться
