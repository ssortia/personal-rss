# Руководство по ведению документации

Этот документ объясняет **как, когда и что** документировать в проектах на базе NexST Template. Он является источником истины для всей команды по вопросам документации.

---

## Философия

> Код отвечает на вопрос «как». Документация отвечает на вопрос «почему».

Три принципа:

1. **Документация — часть задачи.** Фича считается завершённой, когда написан код, тесты и документация. Не раньше.
2. **Минимализм.** Лучше одна актуальная страница, чем десять устаревших. Не документируй очевидное.
3. **Близость к коду.** Документация живёт в репозитории рядом с кодом, обновляется в том же PR.

---

## Структура документации

```
docs/
├── DOCUMENTATION.md       ← этот файл (мета-документ)
├── issues.md              ← трекинг известных проблем и TODO
├── adr/                   ← Architecture Decision Records
│   ├── README.md          ← индекс ADR + шаблон
│   └── NNN-название.md    ← конкретное решение
└── guides/                ← практические руководства
    ├── getting-started.md ← локальная установка
    ├── development.md     ← ежедневная разработка
    ├── adding-a-module.md ← добавление нового модуля
    └── deployment.md      ← деплой в продакшен
```

### Что куда идёт

| Тип информации | Файл |
|---|---|
| Установка и первый запуск | `docs/guides/getting-started.md` |
| Ежедневный workflow | `docs/guides/development.md` |
| Архитектурное решение (нетривиальное) | `docs/adr/NNN-название.md` |
| Пошаговый рецепт добавления фичи | `docs/guides/` |
| Инструкции для AI-ассистента | `CLAUDE.md` |
| Публичная витрина проекта | `README.md` |
| Известные проблемы, техдолг | `docs/issues.md` |

## Правила обновления документации

### README.md

Обновляй при:
- Добавлении или удалении крупной фичи → обнови раздел «Возможности»
- Изменении версии ключевой зависимости → обнови таблицу стека
- Изменении команд запуска → обнови раздел «Быстрый старт»

Не пиши в README:
- Детали реализации (это в guides/)
- Историю изменений (это в git log / CHANGELOG)
- Внутренние решения (это в ADR)

### CLAUDE.md

Обновляй при:
- Изменении архитектуры (новый адаптер, смена ORM, новый пакет в monorepo)
- Изменении команд или скриптов
- Добавлении нового пакета в workspace

`CLAUDE.md` читает AI-ассистент перед началом работы. Он должен быть точным и актуальным — устаревшие инструкции приведут к неверным решениям.

### docs/guides/

Обновляй при:
- Изменении шагов setup (новая env-переменная, новый Docker-сервис)
- Изменении процесса добавления модулей
- Изменении процесса деплоя

Каждый guide должен быть самодостаточным: читатель не должен прыгать между файлами, чтобы выполнить описанную задачу.

## ADR (Architecture Decision Records)

### Когда писать ADR

Пиши ADR если решение:
- **Нетривиально** — не очевидный выбор, были альтернативы
- **Долгосрочное** — будет влиять на проект больше месяца
- **Удивит нового разработчика** — тот, кто придёт позже, должен понять, почему именно так

Примеры решений, требующих ADR:
- Выбор ORM (Prisma vs TypeORM vs Drizzle)
- Выбор адаптера HTTP (Fastify vs Express)
- Стратегия аутентификации (JWT vs sessions vs OAuth-only)
- Монорепо vs полирепо
- Tailwind v3 vs v4

Примеры того, что **не требует** ADR:
- Выбор названия переменной
- Добавление утилитарной функции
- Обновление patch-версии зависимости

### Формат ADR

Имя файла: `NNN-короткое-название-через-дефис.md`, где `NNN` — порядковый номер (001, 002, ...).

```markdown
# NNN. Название решения

**Статус:** Принято | На рассмотрении | Отклонено | Заменено → [ADR-NNN](./NNN-название.md)

**Дата:** YYYY-MM-DD

## Контекст

Что происходит? Какая проблема решается? Какие ограничения существуют?
Пишем факты, без оценок.

## Рассмотренные варианты

- **Вариант A**: краткое описание, плюсы и минусы
- **Вариант B**: краткое описание, плюсы и минусы

## Решение

Что именно выбрали и почему. Конкретно, без воды.

## Последствия

### Положительные
- ...

### Отрицательные / компромиссы
- ...

### Риски
- ...
```

### Как обновить ADR

ADR **не редактируется** после принятия. Если решение изменилось:
1. Создай новый ADR с новым решением
2. В старом ADR измени статус на «Заменено → [ADR-NNN](./NNN-название.md)»

## Руководства (guides/)

### Структура guide

```markdown
# Название (что делаем)

**Цель:** одно предложение — зачем это нужно.

## Предварительные требования

- что должно быть установлено/настроено

## Шаги

### 1. Название шага
...

### 2. Название шага
...

## Проверка

Как убедиться, что всё работает.

## Частые проблемы

| Проблема | Причина | Решение |
|---|---|---|
```

### Правила написания guide

- Каждый шаг — конкретное действие с командой или примером кода
- Проверяй guide на свежей машине перед публикацией
- Указывай ожидаемый результат после каждого неочевидного шага
- Раздел «Частые проблемы» — обязателен, если guide касается setup'а

## Комментарии в коде

### Когда писать комментарий

**Пиши:** когда объясняешь «почему», а не «что».

```typescript
// bcrypt намеренно медленный — это защита от брутфорса.
// Не заменять на SHA-256 ради производительности.
const hash = await bcrypt.hash(password, 10);
```

**Не пиши:** когда код говорит сам за себя.

```typescript
// Плохо: получаем пользователя по id
const user = await this.usersService.findById(id);
```

### Когда НЕ нужен JSDoc

JSDoc-комментарии (`/** */`) нужны только для:
- Публичных методов библиотек/пакетов, потребляемых снаружи
- Не очевидных параметров с неясными типами

Для внутренней логики сервисов и компонентов JSDoc не нужен — типы TypeScript и правильные имена достаточны.

## Для проектов на базе этого шаблона

Если ты используешь NexST Template как основу для своего проекта:

### Что оставить как есть
- Структуру `docs/` и её конвенции
- Формат ADR
- Правила обновления CLAUDE.md

### Что заменить
- `README.md` — перепиши под свой проект (название, описание, фичи)
- `docs/guides/getting-started.md` — обнови под свои env-переменные и сервисы
- `docs/adr/` — можно оставить первые 4 ADR как историю, добавлять свои поверх

### Что добавить
- ADR для каждого нетривиального решения, принятого в проекте
- Guide для каждого специфичного процесса (например, «как добавить новый сервис», «как настроить отправку email»)
- Раздел в README о бизнес-контексте проекта

### Что не делать
- Не удаляй docs/, даже если кажется, что «и так понятно»
- Не дублируй информацию между файлами — ссылайся
- Не пиши в README детали реализации — это в guides/

## Чеклист перед слиянием PR

- [ ] Если добавлена новая фича — обновлён README (раздел «Возможности»)
- [ ] Если изменена архитектура — создан или обновлён ADR
- [ ] Если изменён процесс setup/деплоя — обновлён соответствующий guide
- [ ] Если изменены команды — обновлён CLAUDE.md
- [ ] Комментарии в коде объясняют «почему», а не «что»
